# Отчёт: трассировка системных вызовов с помощью ftrace

**Тема:** написать простую программу на C, настроить **ftrace**, собрать данные
о системных вызовах и проанализировать результаты.

**Дата:** 2026‑06‑24

---

## 1. Цель и требования

| Требование | Где выполнено |
|------------|---------------|
| Программа на C с ≥ 5 различными системными вызовами | `syscall_demo.c` — **11** различных вызовов |
| Корректно настроен и использован ftrace | `trace.sh` — настройка через tracefs, два режима трассировки |
| Подробный анализ результатов | разделы 5–7 этого отчёта |
| Аккуратное оформление кода и отчёта | `Makefile`, комментарии, `README.md`, структура каталога |

---

## 2. Программа на C

`syscall_demo.c` выполняет **детерминированную** последовательность операций,
каждая из которых соответствует отдельному системному вызову. Детерминированность
важна: трасса получается воспроизводимой и легко поддаётся разбору.

| № | Вызов в коде (libc) | Системный вызов (ядро, arm64) | Назначение |
|---|---------------------|-------------------------------|------------|
| 1 | `getpid()`   | `getpid`      | получить PID процесса |
| 2 | `uname()`    | `newuname`    | сведения о ядре |
| 3 | `open()`     | `openat`      | создать/открыть файл |
| 4 | `write()`    | `write`       | записать данные в файл |
| 5 | `fsync()`    | `fsync`       | сбросить буферы на носитель |
| 6 | `lseek()`    | `lseek`       | переместить смещение в файле |
| 7 | `read()`     | `read`        | прочитать данные обратно |
| 8 | `close()`    | `close`       | закрыть дескриптор |
| 9 | `stat()`     | `newfstatat`  | метаданные файла |
| 10| `unlink()`   | `unlink`→`unlinkat` | удалить файл |
| 11| `nanosleep()`| `nanosleep`   | пауза 20 мс |

Программа собирается **статически** (`make`, флаг `-static`). Это сделано
намеренно — обоснование в разделе 6.

## 3. Среда выполнения (важно)

Работа выполнялась на **macOS (Apple Silicon, arm64)**. У ftrace есть жёсткое
ограничение: это **подсистема ядра Linux**, её нельзя запустить на ядре macOS.
Более того, окружение, в котором изначально шла работа (микро‑VM на базе
libkrun/microsandbox), использует урезанное ядро Linux 6.12, собранное **без**
`CONFIG_FTRACE`. Это было проверено напрямую:

```
$ grep -E "tracefs|debugfs" /proc/filesystems
nodev   debugfs                 # tracefs отсутствует
$ mount -t tracefs nodev /sys/kernel/tracing
mount: unknown filesystem type 'tracefs'
```

Чтобы получить **настоящие** данные ftrace, была поднята отдельная одноразовая
виртуальная машина с полноценным ядром Linux:

* **ОС:** Fedora CoreOS 44 (через `podman machine`, провайдер Apple HVF);
* **Ядро:** Linux **7.0.9-205.fc44.aarch64** с включённым ftrace;
* **Доступно трассировщиков:** `function`, `function_graph`, `blk`,
  `wakeup`, `wakeup_rt`, `osnoise`, `timerlat`, `hwlat`, `nop`;
* **Событий системных вызовов:** 324 пары `sys_enter_*/sys_exit_*`.

Весь прогон автоматизирован скриптом `run_demo.sh`: он создаёт VM, собирает
программу в контейнере (в CoreOS нет компилятора), запускает `trace.sh` уже
**внутри** VM, где есть настоящий `tracefs`, и выгружает результаты на хост.
Никакие основные настройки пользователя при этом не затрагиваются — VM
изолирована и удаляется после работы.

> На любой обычной Linux‑машине (Ubuntu, Fedora) с правами root шаги из
> `trace.sh` работают напрямую, без всей этой обвязки с VM.

## 4. Что такое ftrace и как он настроен

**ftrace** (function tracer) — встроенный в ядро Linux трассировщик. Управление
идёт через виртуальную ФС **tracefs**, обычно смонтированную в
`/sys/kernel/tracing`. Запись в «файлы» этого каталога настраивает трассировку,
чтение `trace` — отдаёт собранный буфер.

В работе использованы два независимых режима (см. `trace.sh`).

### 4.1. События системных вызовов (`events/syscalls`)

Это статические точки трассировки (tracepoints) на входе и выходе каждого
системного вызова. Настройка:

```sh
echo nop > current_tracer            # отдельный трассировщик не нужен
echo 1   > events/syscalls/enable    # включить все sys_enter_*/sys_exit_*
```

### 4.2. Фильтрация по процессу (`set_event_pid`)

Чтобы в буфер не попадал системный «шум» от других процессов, трассировка
ограничена ровно нашей программой. Используется приём с `exec`:

```sh
bash -c 'echo $$ > set_event_pid;   # записать СВОЙ PID в фильтр
         echo 1  > tracing_on;       # включить запись
         exec ./syscall_demo'        # exec сохраняет PID -> фильтр продолжает действовать
```

### 4.3. `function_graph` — граф вызовов внутри ядра

Этот трассировщик показывает дерево вызовов функций ядра и **длительность**
каждой. Ограничение по процессу — через `set_ftrace_pid`:

```sh
echo function_graph > current_tracer
echo 1 > options/funcgraph-proc      # показывать процесс
bash -c 'echo $$ > set_ftrace_pid; echo 1 > tracing_on; exec ./syscall_demo'
```

Все собранные «доказательства настройки» лежат в `results/00_env.txt`.

## 5. Анализ: трасса системных вызовов

Полная трасса — `results/10_syscalls.trace`. Формат строки на этом ядре:
вход в вызов печатается как `sys_ИМЯ(аргументы)`, выход — как `sys_ИМЯ -> код`.

### 5.1. Частотная сводка

Скрипт `analyze.sh` (аналог `strace -c`, но по данным ftrace) даёт
`results/30_syscall_summary.txt`:

```
Всего перехвачено системных вызовов (событий входа): 23

КОЛ-ВО  СИСТЕМНЫЙ ВЫЗОВ
------  ---------------
     3  rt_sigaction
     2  writev
     2  close
     1  write, unlinkat, set_tid_address, read, openat, newuname,
        newfstatat, nanosleep, lseek, ioctl, getpid, fsync, fcntl,
        exit_group, execve, dup3
```

23 события входа — это **больше**, чем 11 вызовов в нашем коде. Разница и есть
самое интересное в анализе: ftrace показывает реальную картину, включая то, что
делают оболочка, библиотека C и загрузчик процесса.

### 5.2. Разбор последовательности

Трассу удобно разделить на три фазы.

**Фаза A. Обвязка оболочки и `exec` (PID‑фильтр + запуск программы).**
```
sys_write -> 0x2
sys_dup3(oldfd: 0xa, newfd: 1, flags: 0)        # bash переключает дескрипторы
sys_fcntl(...) ; sys_close(...)
sys_rt_sigaction(sig: 2 ...) x3                 # bash восстанавливает обработчики сигналов
sys_execve(filename: ".../syscall_demo", ...)   # замена образа процесса нашей программой
```
Это последние действия процесса‑обёртки `bash -c '... exec ...'`. Поскольку
`exec` сохраняет PID, фильтр `set_event_pid` продолжает действовать — и дальше в
трассе только наша программа.

**Фаза B. Старт C‑runtime и наши вызовы.**
```
sys_set_tid_address(...)                        # инициализация libc (потоки/futex)
sys_getpid()                  -> 0x92d          # (1) наш getpid(); 0x92d = 2349
sys_ioctl(fd: 1, cmd: 0x5413) -> ...ffe7        # libc: проверка, терминал ли stdout
sys_writev(fd: 1, ..., vlen: 2) -> 0x12         # printf "[demo] pid = 2349" (18 байт)
sys_newuname(...)             -> 0x0            # (2) uname()
sys_openat(... "/tmp/ftrace_demo.txt",
           flags: O_RDWR|O_CREAT|O_TRUNC ...) -> 0x3   # (3) open() -> fd 3
sys_write(fd: 3, buf: ... "ftrace syscall demo: hello, kernel!.",
          count: 0x24)        -> 0x24           # (4) write() 36 байт
sys_fsync(fd: 3)              -> 0x0            # (5) fsync()
sys_lseek(fd: 3, offset: 0, whence: 0) -> 0x0  # (6) lseek() в начало
sys_read(fd: 3, ..., count: 0x7f) -> 0x24      # (7) read() прочитал 36 байт
sys_close(fd: 3)             -> 0x0             # (8) close()
sys_newfstatat(... "/tmp/ftrace_demo.txt" ...) -> 0x0   # (9) stat()
sys_unlinkat(... "/tmp/ftrace_demo.txt" ...)   -> 0x0   # (10) unlink()
sys_nanosleep(rqtp: ...) -> 0x0                # (11) nanosleep(20 мс)
```

**Фаза C. Завершение.**
```
sys_writev(fd: 1, ..., vlen: 2) -> 0xba         # сброс остатка буфера printf (186 байт)
sys_exit_group(error_code: 0)                   # выход всей группы потоков
```

### 5.3. Выводы из разбора (то, чего не видно в исходнике)

1. **`printf` — это не `write`, а `writev`.** Стандартный вывод буферизуется.
   Поскольку stdout не терминал (видно по `ioctl(TIOCGWINSZ) -> 0xffff...ffe7`,
   т.е. `-ENOTTY`), libc включает полную буферизацию и сбрасывает буфер
   векторной записью `writev` всего **2 раза** (18 + 186 = 204 байта — ровно
   объём `prog_stdout.txt`), а не на каждый `printf`.

2. **Имена вызовов в ядре отличаются от имён в libc.** `open()` → `openat`,
   `stat()` → `newfstatat`, `unlink()` → `unlinkat`, `uname()` → `newuname`.
   Современная libc вызывает «*at»‑варианты с базовым каталогом `AT_FDCWD`
   (в трассе это `dfd: 0xffffffffffffff9c` = ‑100).

3. **Аргументы и коды возврата реальны и читаемы.** ftrace декодировал флаги
   `open` как `O_RDWR|O_CREAT|O_TRUNC|O_LARGEFILE`, показал имя файла, число
   байт (`count: 0x24` = 36) и даже первые байты буфера. `write`/`read`
   вернули `0x24` — записано и прочитано одинаковое число байт.

4. **Есть «невидимые» вызовы**, которых нет в нашем коде: `set_tid_address`
   (старт libc), `ioctl` (проверка терминала), `exit_group` (завершение).

## 6. Анализ: тайминги

Метки времени в трассе позволяют измерить длительности (расчёт — встроенным
`awk`, см. ниже):

| Участок | Время |
|---------|-------|
| `execve` (загрузка образа программы) | **0.085 мс** |
| `openat` (создание файла) | **0.108 мс** — самый «дорогой» одиночный вызов |
| `nanosleep` (заказано 20 мс) | **20.223 мс** |
| Вся трассируемая область | **20.507 мс** |
| Из них реальная работа на CPU (без сна) | **≈ 0.284 мс** |

Главное наблюдение: **99 %** времени жизни программы — это сон в `nanosleep`,
а вся полезная работа с файлом заняла менее 0.3 мс. В трассе пауза видна как
разрыв меток времени:

```
67.978307: sys_nanosleep(rqtp: ...)
67.998530: sys_nanosleep -> 0x0        # +20.223 мс
```

Почему 20.223, а не ровно 20.000 мс? `nanosleep` гарантирует **минимальную**
задержку; реальное пробуждение происходит на ближайшем тике/срабатывании
таймера, плюс время на перепланирование — отсюда лишние ~0.2 мс. Механизм этой
паузы хорошо виден в `function_graph` (раздел 7, фрагмент C).

Зачем нужна была статическая сборка (`-static`): у динамического бинарника
старт сопровождается десятками вызовов `openat`/`mmap`/`mprotect` от загрузчика
`ld.so` (подгрузка `libc.so` и т.п.). Статическая сборка убирает этот шум, и в
трассе остаются почти только «наши» вызовы — анализ нагляднее.

## 7. Анализ: function_graph (внутренности ядра)

Файл `results/20_function_graph.trace` (17 678 строк) показывает, что
происходит **внутри** ядра при обработке вызовов. Ключевые выдержки —
`results/40_function_graph_excerpts.txt`.

**Фрагмент A — путь системного вызова `getpid` на arm64.** Любой syscall на
arm64 входит в ядро через исключение `svc` и обработчик `el0_svc`:

```
  do_el0_svc() {
    el0_svc_common.constprop.0() {
      invoke_syscall.constprop.0() {
        __arm64_sys_getpid() {
          __task_pid_nr_ns() {
            __rcu_read_lock();
            __rcu_read_unlock();
          }   0.708 us
        }     0.958 us
      }       1.625 us
    }         1.875 us
  }           2.167 us
```
Видно и диспетчеризацию (`invoke_syscall`), и то, что получение PID защищено
RCU‑секцией. Весь вызов — 2.167 мкс.

**Фрагмент B — `write` в файл на tmpfs: слой VFS и проверка безопасности.**
```
__arm64_sys_write() {
  ksys_write() {
    fdget_pos();
    vfs_write() {
      rw_verify_area() {
        security_file_permission() {
          selinux_file_permission() { avc_policy_seqno(); }   # проверка прав SELinux
        }
      }
      shmem_file_write_iter() { ... }     # /tmp — это tmpfs (shmem)
```
Хорошо видно, что **простой `write()` — это не «просто запись»**: ядро проходит
обобщённый слой VFS (`vfs_write`), проверяет права через LSM/SELinux
(`security_file_permission`), и лишь затем вызывает реализацию конкретной ФС
(`shmem_*`, т.к. `/tmp` смонтирован как tmpfs).

**Фрагмент C — `nanosleep`: постановка таймера и уход в сон.**
```
__arm64_sys_nanosleep() {
  hrtimer_nanosleep() {
    do_nanosleep() {
      hrtimer_start_range_ns() { ... enqueue_hrtimer(); }
      schedule() { ... }          # процесс снимается с CPU до срабатывания таймера
```
Именно `schedule()` объясняет разрыв в 20 мс из раздела 6: процесс ставит
hrtimer и добровольно уступает CPU, пока таймер не разбудит его.

## 8. Выводы

1. Простая программа из 11 строк‑вызовов на C порождает **23** системных вызова
   на уровне ядра — за счёт работы оболочки, libc и завершения процесса.
2. ftrace, настроенный через `tracefs`, дал две взаимодополняющие картины:
   * **события `syscalls`** — *что* за вызовы, с какими аргументами и кодами
     возврата (раздел 5);
   * **`function_graph`** — *как* эти вызовы устроены внутри ядра и сколько
     длятся (раздел 7).
3. Анализ выявил неочевидные вещи: `printf`→`writev` с буферизацией, подмену
   `open`/`stat`/`unlink` на `*at`‑варианты, проход `write` через VFS+SELinux и
   механизм сна через hrtimer + `schedule`.
4. Фильтрация по PID (`set_event_pid`/`set_ftrace_pid`) и статическая сборка
   позволили получить чистую, воспроизводимую трассу почти без постороннего шума.

## 9. Как воспроизвести

**На любом Linux с ftrace (Ubuntu/Fedora), от root:**
```sh
make
sudo ./trace.sh
./analyze.sh results/10_syscalls.trace
```

**В этой среде (macOS) — полностью автоматически в одноразовой VM:**
```sh
./run_demo.sh                 # поднимет VM, соберёт, оттрассирует, заберёт results/
podman machine rm -f ftrace-vm   # убрать VM после работы
```
